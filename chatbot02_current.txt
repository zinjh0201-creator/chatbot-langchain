✦ 프로젝트의 전체 구조와 핵심 로직을 분석한 결과입니다.

  1. 프로젝트 전체 구조 및 기술 스택

  이 프로젝트는 FastAPI(백엔드)와 React(프론트엔드)를 결합한 전형적인 모던 웹 애플리케이션 구조를 가지고 있으며, LangChain을 활용하여 RAG(Retrieval-Augmented Generation) 기능을 구현했습니다.

   * Frontend: React (TypeScript), Vite, CSS (Vanilla)
   * Backend: Python (FastAPI), LangChain, PyPDF
   * Database: Supabase (PostgreSQL + pgvector 확장), asyncpg 드라이버 사용
   * AI Models: Google Gemini (Chat: gemini-1.5-flash 등, Embeddings: text-embedding-0.4)

  ---

  2. 프론트엔드-백엔드 통신 방식

  프론트엔드(App.tsx)와 백엔드(main.py)는 REST API를 통해 통신하며, 특히 채팅 기능은 실시간성을 위해 SSE(Server-Sent Events) 기반의 스트리밍 방식을 사용합니다.

   * 주요 API 엔드포인트:
       * GET /health: 서버 상태 확인
       * GET /documents: 업로드된 문서 목록 조회
       * POST /chat-stream: SSE를 통한 RAG 기반 스트리밍 채팅
       * POST /ingest-pdf: PDF 파일을 업로드하고 벡터화하여 DB에 저장
   * 특이 사항: 프론트엔드에서 fetch의 ReadableStream을 사용하여 백엔드에서 보내는 JSON 데이터 조각들을 실시간으로 UI에 반영합니다.

  ---

  3. LangChain (RAG) 로직 구현 방식

  RAG 파이프라인은 backend/app/langchain_rag.py에 집중되어 있으며, LCEL(LangChain Expression Language)을 사용하여 선언적으로 구성되었습니다.

   1. Ingestion (문서 등록): main.py에서 PDF를 페이지별로 읽고, 약 800자 단위로 청크를 나눕니다. Gemini Embedding API를 통해 벡터로 변환한 후 Supabase에 저장합니다.
   2. Retrieval (검색): 사용자의 질문이 들어오면 질문을 벡터로 변환하고, DB에서 코사인 유사도(<=> 연산자)를 기준으로 유사한 문서를 찾습니다. (유사도 0.8 이상 필터링 적용)
   3. Augmentation & Generation (생성):
       * Document 모드: 검색된 문서가 있으면 해당 내용을 컨텍스트로 프롬프트에 삽입하여 답변을 생성합니다.
       * Gemini 모드: 검색된 문서가 없거나 유사도가 낮으면 Gemini의 일반 지식을 기반으로 답변합니다.
   4. Memory (대화 기억): RunnableWithMessageHistory를 사용하여 세션별 대화 기록을 관리하며, 프롬프트 팽창을 막기 위해 최근 3개의 메시지만 컨텍스트로 유지하는 전략을 사용합니다.

  4. 코드 개선이 필요한 문제점 및 제안

  분석 결과, 다음과 같은 측면에서 개선이 필요해 보입니다.

  A. 구조 및 효율성 측면
   1. 중복된 임베딩 로직: main.py에서는 gemini_client.py의 함수를 쓰고, langchain_rag.py에서는 LangChain의 GoogleGenerativeAIEmbeddings를 사용합니다. 설정이 달라질 경우 데이터 불일치가 발생할 수 있으므로 하나로 통합해야 합니다.
   2. 수동 청크 분할: main.py에서 문자열 슬라이싱 방식으로 청크를 나누고 있습니다. 문맥 유실을 방지하기 위해 LangChain의 RecursiveCharacterTextSplitter를 사용하는 것이 좋습니다.
   3. PDF 처리 속도: PDF 인제스션 시 청크를 하나씩 DB에 INSERT하고 있습니다. executemany 등을 사용하여 Bulk Insert로 처리하면 대용량 문서 처리 속도가 대폭 향상됩니다.

  B. 보안 측면
   1. CORS 설정: 현재 CORSMiddleware의 설정이 settings.cors_origins_list를 따르고 있으나, 운영 환경에서 와일드카드(*)가 허용되지 않도록 엄격히 관리해야 합니다.
   2. 환경 변수 검증: config.py에서 API 키 등이 누락되었을 때의 예외 처리가 필요합니다.

  C. 기능 및 확장성 측면
   1. 메모리 휘발성: 현재 대화 기록(message_histories)이 서버 메모리(dict)에 저장됩니다. 서버 재시작 시 기록이 사라지므로, Redis나 DB(PostgreSQL)에 페르시스턴트(Persistent)하게 저장하는 구조로 변경해야 합니다.
   2. 에러 핸들링: PDF 파싱 실패나 임베딩 API 타임아웃 시 사용자에게 더 상세한 가이드(예: "텍스트가 없는 PDF입니다" 등)를 제공할 수 있도록 예외 처리를 세분화해야 합니다.

  어떤 부분부터 개선을 시작해 볼까요? 우선순위가 높은 메모리 영속화나 청크 분할 로직 최적화부터 진행하는 것을 추천드립니다.
